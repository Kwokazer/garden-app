# Отчет о реорганизации сервисных классов

## 1. Анализ исходной ситуации

В проекте были идентифицированы следующие сервисные классы:

1. `AuthService` - управление аутентификацией и авторизацией (`auth_service.py`)
2. `UserService` - управление пользователями (`user_service.py`)
3. `PlantService` - управление растениями (`plant_service.py`)
4. `PlantCategoryService` - управление категориями растений (`plant_category_service.py`)
5. `ClimateZoneService` - управление климатическими зонами (`climate_zone_service.py`)
6. `QAService` - управление вопросами и ответами (объединенный сервис) (`qa_service.py`)
7. `PlantSearchService` - поиск растений (`search_service.py`) - несогласованное наименование файла и класса

## 2. Выявленные проблемы

1. **Структурные проблемы**:
   - `QAService` содержит слишком много ответственностей, нарушая принцип Single Responsibility
   - Смешивание бизнес-логики для вопросов и ответов в одном классе
   - Проблемы с импортами в `QAService` (неправильные пути для репозиториев)
   - Несогласованное наименование файла `search_service.py` для класса `PlantSearchService`

2. **Проблемы с именованием**:
   - Несоответствие именования методов между разными сервисами
   - Методы с техническими названиями в `PlantSearchService`: `search_plants`, `search_categories` вместо `get_plants`, `get_categories`
   - Некоторые методы имеют технические названия вместо отражающих бизнес-операции

3. **Архитектурные проблемы**:
   - Прямая работа с SQL в некоторых сервисах вместо использования репозиториев
   - Не вполне последовательная обработка ошибок
   - Недостаточное абстрагирование уровня бизнес-логики от деталей реализации

## 3. План реорганизации

1. **Разделение QAService на два специализированных сервиса**:
   - `QuestionService` - управление вопросами
   - `AnswerService` - управление ответами

2. **Переименование файлов**:
   - `search_service.py` → `plant_search_service.py` для соответствия имени класса

3. **Унификация именования методов**:
   - Методы получения: `get_*`
   - Методы создания: `create_*`
   - Методы обновления: `update_*`
   - Методы удаления: `delete_*`
   - Специфические операции: `vote_for_*`, `mark_as_*`

4. **Стандартизация обработки ошибок**:
   - Использование стандартных классов исключений из `base.py`
   - Корректное логирование ошибок
   - Четкое разделение обязанностей между сервисами и API

5. **Обновление обработчиков API для работы с новыми сервисами**:
   - `endpoints/questions.py` - использование `QuestionService`
   - `endpoints/answers.py` - использование `AnswerService`
   - `endpoints/plants.py` - использование переименованных методов

## 4. Выполненные изменения

1. **Исправление импортов репозиториев**:
   - Все импорты репозиториев теперь используют корректные пути:
     ```python
     from app.infrastructure.database.repositories.question_repository import QuestionRepository
     from app.infrastructure.database.repositories.answer_repository import AnswerRepository
     ```

2. **Создание новых сервисных классов**:
   - Создан `QuestionService` (question_service.py)
   - Создан `AnswerService` (answer_service.py)
   - Обновлен файл `__init__.py` для экспорта новых сервисов

3. **Переименование файлов и методов**:
   - Переименован файл `search_service.py` в `plant_search_service.py`
   - Методы в `PlantSearchService`:
     - `search_plants` → `get_plants`
     - `search_categories` → `get_categories`

4. **Обновление API точек входа**:
   - Обновлены импорты и использование сервисов в `endpoints/questions.py`
   - Обновлены импорты и использование сервисов в `endpoints/answers.py`
   - Обновлен файл `endpoints/plants.py` для работы с переименованными методами

5. **Стандартизация кода**:
   - Унифицированы наименования методов
   - Улучшены сообщения об ошибках
   - Улучшена обработка исключений

## 5. Рекомендации для дальнейшего улучшения

1. **Расширение использования дженериков**:
   - Базовый класс `BaseService` можно расширить для включения общих CRUD операций с использованием дженериков

2. **Улучшение кэширования**:
   - Разработать единый подход к кэшированию для всех сервисов
   - Создать абстракцию над Redis для более гибкого управления кэшем

3. **Устранение прямых SQL-запросов**:
   - Заменить прямые SQL-запросы в методах голосования на использование методов репозиториев
   - Добавить соответствующие методы в репозитории для работы с голосованием

4. **Усиление валидации**:
   - Добавить более строгую валидацию входных данных на уровне сервисов
   - Создать отдельные валидаторы для сложных бизнес-правил

5. **Улучшение логирования**:
   - Внедрить структурированное логирование
   - Добавить контекстное логирование с идентификаторами запросов

6. **Тестирование**:
   - Разработать комплексные тесты для всех сервисов
   - Внедрить фиктивные реализации репозиториев для тестирования бизнес-логики в изоляции
